# 教学教案：AI Agent RAG 系统实战

教学对象：希望通过一个完整项目掌握 AI Agent 应用开发的学习者。

教学目标：带你从零实现一套具备工程落地能力的 AI Agent RAG 系统，并能讲清楚技术选型、系统架构、核心实现和优化思路。

教学方式：

- 每次只讲当前阶段必须理解的知识。
- 每次学习都要落到代码。
- 每个模块都有验收标准。
- 遇到复杂概念时，先用业务场景解释，再进入代码。
- 不追求一次性写完所有功能，按可验证的小版本推进。
- 如果学习者没有明确要求代写或实现代码，教学者只讲解实现方法、步骤和要点，不自动修改项目代码。

## 1. 总体教学路线

```text
项目理解
  -> 工程骨架
  -> 数据库建模
  -> Vue 聊天工作台
  -> 文档入库
  -> RAG 检索
  -> 流式问答
  -> 图片资产召回
  -> LangGraph Multi-Agent
  -> 工程化和面试表达
```

## 2. 教学原则

### 2.1 先业务，后技术

每个技术点都必须回答：

- 它解决什么业务问题？
- 为什么这里需要它？
- 不用它会有什么问题？
- 它在项目中对应哪段代码？

### 2.2 先主链路，后增强

先完成能跑通的核心链路：

```text
用户提问 -> 检索知识库 -> LLM 回答 -> 前端展示
```

再增强：

- 父子 chunk。
- 图片召回。
- rerank。
- Multi-Agent。
- 停止生成。
- Agent 轨迹。

### 2.3 每节课必须有产出

每节课结束时至少完成一种产出：

- 一个可运行接口。
- 一个数据库模型。
- 一个前端页面。
- 一个可执行脚本。
- 一个测试。
- 一个 README 或设计说明。

### 2.4 默认教学，不默认代写

本项目默认采用教学模式：

- 学习者没有明确要求“帮我写代码”时，教学者不自动创建、修改或删除项目代码。
- 教学者应先讲清楚实现目标、设计原因、数据流、文件位置和编码顺序。
- 教学者可以给出小段示例代码，但示例用于解释概念，不默认写入文件。
- 学习者需要自己动手实现关键代码，并在遇到问题时让教学者讲解、检查或排错。
- 当学习者明确授权实现某个模块时，教学者才进入代码修改模式，并在修改前说明改动范围。

这个原则的目的是保证学习者真正掌握系统，而不是只得到一套看起来能运行的代码。

## 3. 阶段一：项目认知与工程骨架

教学目标：

- 理解整个系统要做什么。
- 理解 FastAPI + Vue + PostgreSQL + Redis 的职责边界。
- 搭建后端和前端基础工程。

讲解要点：

- 什么是 AI Agent 应用，不只是调用大模型 API。
- FastAPI 在系统中的位置。
- Vue 在系统中的位置。
- PostgreSQL 和 Redis 的职责区别。
- 为什么当前阶段不做前后端分离部署。

实操任务：

- 初始化 `pyproject.toml`。
- 创建 `app/main.py`。
- 创建 `GET /api/health`。
- 初始化 `frontend/`。
- 创建 Vue 首页。
- 配置 FastAPI 挂载 Vue 静态资源的预留结构。

验收标准：

- 后端能启动。
- 前端能启动。
- 健康检查接口返回正常。

我会重点教你：

- 如何看一个后端项目结构。
- 如何判断代码应该放在 API、service、model 还是 schema。
- 如何避免一开始就把逻辑写乱。

## 4. 阶段二：数据库建模

教学目标：

- 掌握 AI Agent RAG 系统的数据结构设计。
- 理解知识库、文档、chunk、图片资产、会话、消息、任务之间的关系。

讲解要点：

- 业务数据和向量数据如何一起存。
- 为什么文档和 chunk 要拆表。
- 为什么图片要有 `document_assets` 表。
- 为什么 Agent 任务需要 `agent_task_events`。
- 普通 chunk 和父子 chunk 的表结构差异。

实操任务：

- 建立 SQLAlchemy base。
- 建立数据库 session。
- 建立模型：
  - `KnowledgeBase`
  - `Document`
  - `DocumentChunk`
  - `DocumentAsset`
  - `Conversation`
  - `Message`
  - `AgentTask`
  - `AgentTaskEvent`
- 配置 Alembic。
- 生成第一版迁移。

验收标准：

- 数据库表能创建。
- 能插入一条知识库记录。
- 能插入一条会话和消息记录。

我会重点教你：

- 表设计怎么从业务流程推导出来。
- 字段不是越多越好，哪些字段第一版必须有。
- JSONB、外键、自关联、索引分别用在哪里。

## 5. 阶段三：Vue 聊天工作台基础

教学目标：

- 做出可用的聊天产品界面。
- 理解会话、消息、生成状态在前端如何管理。

讲解要点：

- 主流大模型聊天界面的信息结构。
- 为什么左侧会话列表和中间消息列表要分组件。
- Pinia 如何管理跨组件状态。
- 生成中状态、停止按钮、编辑模式如何设计。

实操任务：

- 创建 `ChatWorkspace.vue`。
- 创建 `ConversationSidebar.vue`。
- 创建 `MessageList.vue`。
- 创建 `MessageItem.vue`。
- 创建 `ChatInput.vue`。
- 创建 conversation store。
- 创建 chat store。
- 接入 mock 数据。

验收标准：

- 页面能展示历史会话。
- 点击会话能切换消息。
- 输入框能发送一条本地 mock 消息。

我会重点教你：

- 前端组件如何拆。
- 状态应该放组件里还是 store 里。
- 如何避免聊天页面变成一个巨大组件。

## 6. 阶段四：会话和消息接口

教学目标：

- 让前端历史会话接入真实后端。
- 理解会话和消息的 CRUD。

讲解要点：

- RESTful API 设计。
- 请求 schema 和响应 schema。
- service 层如何封装业务逻辑。
- 编辑消息和重新生成为什么会影响后续消息。

实操任务：

- `GET /api/conversations`
- `POST /api/conversations`
- `GET /api/conversations/{id}`
- `PATCH /api/conversations/{id}`
- `DELETE /api/conversations/{id}`
- `PATCH /api/messages/{id}`
- 前端接入真实 API。

验收标准：

- 前端能创建会话。
- 前端能加载历史消息。
- 前端能重命名会话。
- 前端能编辑用户消息。

我会重点教你：

- 前端页面和后端接口如何一起演进。
- 编辑消息的第一版简化方案怎么做。

## 7. 阶段五：文档入库和普通 RAG

教学目标：

- 跑通第一条 RAG 主链路。

讲解要点：

- RAG 为什么需要文档解析、chunk、embedding、检索。
- 普通 chunk 的适用场景。
- pgvector 的基本作用。
- 引用来源怎么保存和返回。

实操任务：

- 文档上传接口。
- 文本文档解析。
- 普通 chunk 切分。
- embedding provider 封装。
- chunk 入库。
- query 检索接口。

验收标准：

- 上传一份文本或 Markdown 文档。
- 数据库中生成 chunks。
- 输入 query 可以召回相关 chunk。

我会重点教你：

- chunk size 怎么选。
- embedding 调用应该在哪里封装。
- 检索结果为什么不能直接等同最终回答。

## 8. 阶段六：父子 Chunk

教学目标：

- 掌握更适合长文档的 RAG 结构。

讲解要点：

- 普通 chunk 的问题。
- parent chunk 和 child chunk 的职责。
- 为什么 child 负责召回，parent 负责上下文。
- 父子 chunk 如何去重和回填。

实操任务：

- 增加 chunk strategy 配置。
- 实现 parent splitter。
- 实现 child splitter。
- 存储 `parent_chunk_id`。
- 实现 parent-child retriever。

验收标准：

- 同一份文档可以选择普通 chunk 或父子 chunk 入库。
- query 命中 child 后能回填 parent。
- 最终返回上下文比普通 chunk 更完整。

我会重点教你：

- 为什么父子 chunk 是工程上的折中方案。
- 如何防止回填 parent 后上下文过长。

## 9. 阶段七：Word 图片抽取和图片召回

教学目标：

- 让 RAG 支持大量示例图、流程图、截图类文档。

讲解要点：

- 为什么图片不能只当附件。
- 为什么要把图片位置替换成 URL。
- `document_assets` 如何设计。
- 图片如何通过周围文本参与召回。

实操任务：

- 解析 Word 文档。
- 抽取内嵌图片。
- 保存图片到本地静态目录。
- 生成图片 URL。
- 正文中替换为 `![asset:{asset_id}]({url})`。
- 图片资产入库。
- 基于 `caption + surrounding_text` 召回图片。

验收标准：

- 上传 Word 后可以看到图片资产记录。
- chunk 文本中包含图片 URL 占位符。
- 问到相关示例时，回答结果返回图片 URL。
- 前端能预览图片资产。

我会重点教你：

- 多模态能力还没接入时，如何先用文本上下文召回图片。
- 为什么这比简单丢弃图片更接近真实工程。

## 10. 阶段八：流式问答和停止生成

教学目标：

- 实现接近真实大模型产品的回答体验。

讲解要点：

- 普通 HTTP 响应和流式响应的区别。
- SSE 的基本格式。
- 前端如何追加 token。
- 停止生成为什么需要 run_id。
- Redis 如何存任务取消状态。

实操任务：

- `POST /api/chat/stream`
- SSE 事件格式。
- 前端流式渲染。
- `POST /api/chat/stop`
- Redis 保存运行状态。
- 生成过程中检查取消状态。

验收标准：

- 前端能看到 AI 逐字输出。
- 点击停止后生成中断。
- 已生成内容保留。

我会重点教你：

- 流式输出的数据结构。
- 停止生成的后端控制点。

## 11. 阶段九：LangGraph Multi-Agent

教学目标：

- 用 LangGraph 实现真实工作流，而不是简单链式调用。

讲解要点：

- LangGraph State。
- Node。
- Edge。
- Conditional Edge。
- Agent 任务为什么需要事件记录。
- Reviewer 为什么可以打回 Planner。

实操任务：

- 定义 `AgentState`。
- 实现 Router Node。
- 实现 Retriever Node。
- 实现 Analyzer Node。
- 实现 Planner Node。
- 实现 Reviewer Node。
- 实现 Agent task API。
- 前端展示 Agent 思考过程和执行轨迹。

验收标准：

- 用户提交复杂任务后，系统按节点执行。
- 每个节点有事件记录。
- 前端能看到执行过程。
- 最终返回结构化结果。

我会重点教你：

- Agent 节点不是越多越好。
- 什么逻辑适合放进 LangGraph，什么逻辑留在 service 层。

## 13. 每次教学的固定流程

每次我们推进一个模块时，按这个流程：

1. 说明本次目标。
2. 画出本次模块在系统中的位置。
3. 讲最少必要理论。
4. 写代码。
5. 运行验证。
6. 记录问题。
7. 更新学习记录。
8. 总结本次可用于面试的表达。

## 14. 对你的要求

你每次学习时需要做三件事：

- 跟着实现，不只看解释。
- 遇到不懂的概念及时问。
- 每完成一个模块，用自己的话复述它解决了什么问题。

判断你是否真的学会，不看你记住多少术语，而看你能不能回答：

- 这个模块为什么需要？
- 数据从哪里来，到哪里去？
- 如果它坏了，系统会出现什么问题？
- 怎么测试它？
- 面试时怎么讲？
